# Security Policy

MinSpec takes security seriously, especially anywhere package installation, recipes, AI-assisted tooling, generated files, local development tools, or agent experiments are involved.

---

## Reporting Security Issues

Do not disclose security vulnerabilities publicly before maintainers have had a chance to review them.

Use GitHub private vulnerability reporting where available.

Include:

- affected repository
- affected package/version if known
- reproduction steps
- impact summary
- suggested mitigation if available

---

## Scope

This repository is currently a namespace anchor and future metapackage.

Security concerns may include:

- package namespace control
- Packagist metadata
- dependency declarations
- repository permissions
- release provenance
- GitHub Actions or workflow changes
- supply-chain or maintainer-account compromise

This policy does not replace Symfony's own security reporting process for Symfony framework vulnerabilities.

---

## Source Authority

Do not submit security fixes as unsolicited pull requests during incubation unless explicitly invited.

A security report may include suggested remediation, but trusted changes must still enter through an approved MinSpec source path.
